# Automated API Testing

## Context and Problem Statement

The Oil & Gas Forecast API exposes multiple endpoints that are consumed by external clients. Ensuring API reliability, correctness, and security is critical to guarantee a consistent user experience and prevent regressions during development.

Initially, endpoint behavior was validated manually. Manual testing, however, is error-prone, difficult to reproduce consistently, and becomes increasingly inefficient as the project evolves.

The team needed an automated testing strategy capable of validating endpoint behavior, input validation, business rules, and API authentication mechanisms.

## Decision Drivers

- Improve API reliability and stability.
- Detect regressions early during development.
- Validate business rules automatically.
- Ensure API security through authentication tests.
- Reduce manual testing effort.
- Provide a reusable and maintainable testing framework.

## Considered Options

### Continue executing tests manually

Developers manually execute API requests and validate responses.

### Implement automated tests using Python unittest

Create a repeatable suite of tests executed locally during development.

### Rely only on end-to-end manual validation

Validate the system exclusively through manual API usage after implementation.

## Decision Outcome

The team decided to implement automated tests using Python's `unittest` framework together with FastAPI's `TestClient`.

The tests were initially executed locally during development to ensure that API behavior remained consistent and compliant with expected requirements.

The automated test suite covers functional behavior, input validation, business rules, HTTP compliance, and API authentication.

## Implemented Test Coverage

### Forecast Endpoint (`/forecast`)

The following scenarios were tested:

- Valid requests return HTTP 200.
- Missing required parameters return HTTP 422.
- Invalid parameter formats return HTTP 422.
- Invalid date ranges (`date_end < date_start`) return HTTP 400.
- Equal start and end dates are accepted.
- Unsupported HTTP methods return HTTP 405.
- Non-existent routes return HTTP 404.
- Invalid well identifier formats are rejected.

### Wells Endpoint (`/wells`)

The following scenarios were tested:

- Valid requests return HTTP 200.
- Missing `date_query` returns HTTP 422.
- Invalid date formats return HTTP 422.
- Unsupported HTTP methods return HTTP 405.
- Response structure is validated.

### API Key Security

Security tests verify:

- Valid API keys grant access.
- Invalid API keys return HTTP 403.
- Missing API keys return HTTP 403.
- Custom error messages are correctly returned.

Dependency overrides were used during testing to simplify API key handling.

## Consequences

### Positive Consequences

- Improved API reliability.
- Increased confidence during development.
- Faster detection of defects.
- Better documentation of expected API behavior.
- Reduced manual testing effort.

### Negative Consequences

- Tests require maintenance when API contracts change.
- Additional development time is required to create and update tests.

## Confirmation

The implemented test suite successfully validates endpoint behavior, business rules, and authentication mechanisms. The tests can be executed repeatedly and consistently, reducing the risk of introducing defects into the application.

## Pros and Cons of the Options

### Continue executing tests manually

**Pros**

- No implementation effort.
- Simple for very small projects.

**Cons**

- Error-prone.
- Difficult to reproduce consistently.
- Does not scale as the project grows.

### Implement automated tests using Python unittest

**Pros**

- Repeatable and reliable.
- Integrated into the Python standard library.
- Improves software quality.
- Facilitates future automation.

**Cons**

- Requires implementation and maintenance effort.
- Test coverage must evolve with the application.

### Rely only on end-to-end manual validation

**Pros**

- Simulates real user interactions.

**Cons**

- Slow and difficult to automate.
- High probability of missing edge cases.

## More Information

The test suite was implemented using FastAPI's `TestClient` and Python's `unittest` framework. Continuous execution of these tests through GitHub Actions is documented separately in ADR-005: Continuous Integration with GitHub Actions.
