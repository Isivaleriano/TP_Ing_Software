# API Key Security

## Context and Problem Statement

The project exposes a FastAPI API for oil and gas forecast and wells information. These endpoints should not be accessible without a basic authorization mechanism.

Without access control, any client could call the API endpoints directly and retrieve project data or execute forecast requests. The team needed a simple security mechanism that could be implemented quickly, tested automatically, and reused across endpoints.

The first version of the API required lightweight protection rather than a full user authentication system.

## Decision Drivers

- Prevent unrestricted access to API endpoints.
- Keep the security mechanism simple and easy to test.
- Reuse the same validation logic across multiple routes.
- Avoid adding unnecessary authentication infrastructure for the current scope.
- Return a clear error when access is missing or invalid.
- Support testing through API clients such as Insomnia and automated tests.

## Considered Options

### Option 1: No authentication

Expose the API endpoints without any access control.

**Pros**
- Simplest implementation.
- Easy to test from a browser.
- No headers or credentials required.

**Cons**
- Leaves endpoints publicly accessible.
- Provides no protection against unauthorized requests.
- Does not meet the project security requirement.
- Makes it harder to control API usage.

### Option 2: API key through a custom header

Require clients to send a valid API key in the X-API-Key header.

**Pros**
- Simple to implement in FastAPI.
- Easy to reuse through dependency injection.
- Easy to test with automated tests and API clients.
- Provides a first layer of endpoint protection.
- Does not require user accounts, sessions, or token infrastructure.

**Cons**
- Less secure than OAuth2 or JWT based authentication.
- Key rotation and secret management are limited in the current implementation.
- Clients must explicitly include the header in every request.
- Direct browser testing becomes less convenient.

### Option 3: OAuth2 or JWT authentication

Implement a token based authentication mechanism with users, login, and issued access tokens.

**Pros**
- More robust and extensible.
- Supports user identity and authorization flows.
- Better suited for production systems with multiple users.

**Cons**
- More complex to implement and maintain.
- Requires user management or an identity provider.
- Adds unnecessary complexity for the current project scope.
- Longer implementation and testing effort.

## Decision Outcome

We adopted **API key authentication using the X-API-Key header**.

The validation logic is centralized in:

api/v1/security.py


The API defines a get_api_key dependency that reads the X-API-Key header and compares it against the expected key.

If the key is valid, the request continues. If the key is missing or invalid, the API returns:

HTTP 403 Forbidden


with the following error message:

Access denied. Invalid or missing API key in header.


## Implementation Details

### Security Dependency

The API uses FastAPI's APIKeyHeader security helper with:

X-API-Key


The validation is implemented as a reusable dependency:

get_api_key


This keeps authentication logic in one place and avoids duplicating header validation in each route.

### Protected Endpoints

The following endpoints depend on get_api_key:

- /api/v1/forecast
- /api/v1/wells

Requests to these endpoints are only processed when the client sends a valid API key.

### Testing and API Client Usage

Automated tests validate:

- Valid API keys grant access.
- Missing API keys return 403.
- Invalid API keys return 403.
- The returned error message is consistent.

Because protected endpoints require a custom header, manual endpoint testing is performed through an API client such as Insomnia instead of direct browser access.

## Decision Outcome

The selected approach adds a first layer of access control while keeping the implementation small, centralized, and easy to test.

It is appropriate for the current project scope and can be replaced later by a stronger authentication mechanism if production requirements increase.

### Consequences

#### Positive

- API endpoints are no longer openly accessible.
- Security validation is centralized in a reusable dependency.
- The implementation is simple and testable.
- Forecast and wells endpoints share the same access control mechanism.
- Automated tests verify valid, missing, and invalid API key behavior.

#### Negative

- The current API key is static.
- The implementation does not provide user level authentication or authorization.
- Key rotation is not automated.
- Clients must include the X-API-Key header in every request.
- Browser only testing is less convenient for protected endpoints.
