# Integrating the Machine Learning Pipeline into the Forecast API

## Context and Problem Statement

Phase 3 requires exposing the trained machine learning model through the existing REST API.

The project already included a forecasting endpoint (/forecast) implemented as a mock service during previous phases. The new predictive model had to be integrated without breaking the existing API while allowing the prediction service to retrieve features from the Feature Store and the production model from MLflow.

The main architectural decision was how to integrate machine learning inference into the existing API.

## Decision Drivers

- Preserve compatibility with the endpoints implemented in previous phases.
- Keep machine learning inference isolated from the legacy mock implementation.
- Allow the prediction service to evolve independently.
- Reuse FastAPI's routing and dependency injection.
- Make the endpoint easy to test through unit tests by mocking the prediction layer.

## Considered Options

### Replace the existing /forecast endpoint

Modify the original endpoint so every forecast request uses the machine learning model.

### Create a separate microservice

Expose a completely independent ML API responsible only for predictions.

### Add a dedicated ML endpoint inside the existing API (chosen)

Keep the existing API and add a new endpoint (/forecast/ml) responsible for machine learning inference.

## Decision Outcome

Chosen option: **introduce a dedicated /forecast/ml endpoint**.

The original /forecast endpoint remains unchanged for compatibility with previous phases.

The new endpoint delegates prediction to the prediction service (predict_one() or predict_one_gas(), depending on the requested commodity), which encapsulates the interaction with the Feature Store and the MLflow Model Registry. The API layer is therefore responsible only for:

- validating request parameters;
- handling authentication;
- selecting the requested commodity (oil or gas);
- translating application errors into HTTP responses;
- returning the prediction result.

This keeps business logic outside the HTTP layer and allows the prediction module to evolve independently.

## Consequences

### Positive Consequences

- Existing clients continue using the original endpoint without modification.
- Machine learning inference is isolated from the mock implementation.
- The prediction logic can be tested independently of FastAPI.
- API unit tests can mock the prediction service (predict_one() / predict_one_gas()) without requiring PostgreSQL or MLflow.
- The endpoint follows the same authentication and monitoring mechanisms as the rest of the API.

### Negative Consequences

- Two forecasting endpoints now coexist.
- Some validation and monitoring logic is duplicated between endpoints.

## Pros and Cons of the Options

### Replace /forecast

**Pros**

- Single forecasting endpoint.

**Cons**

- Breaks compatibility with previous project phases.
- Couples the existing API to the ML implementation.

### Separate ML microservice

**Pros**

- Complete separation between API and inference service.

**Cons**

- Additional deployment and networking complexity.
- Unnecessary for the current project size.

### Dedicated /forecast/ml endpoint

**Pros**

- Backward compatible.
- Clear separation between legacy and ML functionality.
- Simple integration into the existing FastAPI project.
- Easy to unit test using mocked predictions.

**Cons**

- Two endpoints provide forecasting functionality.

## More Information

The implementation introduces:

- a new FastAPI endpoint (/forecast/ml);
- a dedicated response model (ForecastMLResponse);
- support for forecasting both oil and gas production through the commodity query parameter;
- the API delegates inference to predict_one() or predict_one_gas(), which encapsulate the interaction with the Feature Store and the MLflow Model Registry;
- unit tests covering successful predictions, invalid requests, unsupported commodities and prediction failures through mocked inference.
