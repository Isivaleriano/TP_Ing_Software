# API Implementation with FastAPI

## Co and Problem Statement

The project started from a base FastAPI example provided in class. That example included generic routes such as /hello, /sum, and /user.

The project requirements, however, were focused on an Oil & Gas Forecast API. The API needed to expose domain specific endpoints for querying available wells and production forecasts, following the technical addendum for the project.

The team needed to replace the generic class example with a versioned API structure that could support future growth, testing, security, documentation, monitoring, and deployment.

## Decision Drivers

- Build a REST API aligned with the Oil & Gas Forecast domain.
- Use a lightweight framework suitable for rapid API development.
- Organize the API using versioned routes.
- Provide clear endpoints for wells and forecasts.
- Support automatic request validation for query parameters.
- Keep the first implementation simple through mock data.
- Enable future integration with security, tests, Docker, and monitoring.

## Considered Options

### Option 1: Keep the original class example

Continue using the generic FastAPI routes provided in the class project.

**Pros**
- Already available and easy to run.
- Useful as a learning reference.
- Minimal implementation effort.

**Cons**
- Does not match the Oil & Gas Forecast domain.
- Does not expose the required wells and forecast endpoints.
- Provides no clear foundation for the project API.
- Would require maintaining irrelevant example routes.

### Option 2: Implement the API with Flask

Build the project API using Flask instead of FastAPI.

**Pros**
- Simple and widely used.
- Flexible for small REST APIs.
- Easy to understand.

**Cons**
- Requires more manual setup for validation and documentation.
- Does not provide the same automatic OpenAPI support by default.
- Less aligned with the class starting point.
- More boilerplate for typed query parameters.

### Option 3: Implement a versioned API with FastAPI

Use FastAPI and organize the project under a versioned API structure.

**Pros**
- Matches the class technology stack.
- Provides automatic validation for query parameters.
- Generates OpenAPI documentation automatically.
- Supports dependency injection for security.
- Works well with testing through TestClient.
- Easy to containerize and monitor.

**Cons**
- Requires reorganizing the initial example project.
- Initial endpoints still need mock data until real integration is available.
- Requires route and module structure decisions early in the project.

## Decision Outcome

We adopted **FastAPI with a versioned /api/v1 route structure**.

The API was reorganized under:

api/v1/


The first version replaced the generic example functionality with two domain specific endpoints:

- /api/v1/forecast
- /api/v1/wells

This provided a clean foundation for the Oil & Gas Forecast API while keeping the implementation simple enough for early testing and iteration.

## Implementation Details

### Application Entry Point

The FastAPI application is defined in:

api/v1/main.py


The application includes routers for the forecast and wells modules using the /api/v1 prefix.

The API title is:

Oil & Gas Forecast API


### Forecast Endpoint

The forecast endpoint is implemented in:

api/v1/forecast/routes.py


Endpoint:

GET /api/v1/forecast


Required query parameters:

- id_well
- date_start
- date_end

The endpoint validates:

- Required parameters.
- Date format through FastAPI date parsing.
- Well identifier format using the WELL-### pattern.
- date_end must be greater than or equal to date_start.
- Unknown wells return 404.

The first implementation returns mock forecast data with fixed production values.

### Wells Endpoint

The wells endpoint is implemented in:

api/v1/wells/routes.py


Endpoint:

GET /api/v1/wells


Required query parameter:

- date_query

The endpoint validates the date format through FastAPI date parsing and returns a mock list of available wells.

### Supporting Capabilities

The API implementation also supports later project requirements:

- API key security through FastAPI dependencies.
- Automated tests using FastAPI TestClient.
- Docker execution with Uvicorn.
- OpenAPI documentation.
- Prometheus metrics through instrumentation and custom endpoint metrics.

## Decision Outcome

The selected approach moved the project from a generic class example to a domain specific API aligned with the Oil & Gas Forecast requirements.

It provides a maintainable structure for future development while allowing the team to validate endpoint contracts before integrating real production data.

### Consequences

#### Positive

- The API structure matches the project domain.
- Routes are versioned under /api/v1.
- FastAPI provides automatic request validation and documentation.
- Forecast and wells endpoints have clear contracts.
- Mock data enables early testing before full data integration.
- The structure supports security, monitoring, Docker, and CI testing.

#### Negative

- The first implementation still relies on hardcoded mock data.
- Forecast logic is not yet connected to real production models.
- Some example functionality remains available only for basic health/testing purposes.
- Future changes may be needed when real data replaces mocks.
