# Email Alerting with Grafana

## Context and Problem Statement

The project includes a monitoring stack based on Prometheus and Grafana to observe API availability, latency, traffic, and infrastructure metrics.

Dashboards provide visibility into the system, but they require manual inspection. This is not enough for operational incidents such as service outages, high error rates, or sustained latency degradation.

The team needed an automated alerting mechanism that could notify stakeholders when relevant problems occur, while avoiding exposure of sensitive credentials in the repository.

Initially, email alerting was evaluated through Alertmanager using an alertmanager.yml configuration file. However, that approach required defining Gmail SMTP credentials directly in configuration, creating a risk of leaking secrets through version control.

## Decision Drivers

- Notify the team automatically when critical API issues occur.
- Avoid storing email credentials in the repository.
- Keep alerting configuration reproducible and version-controlled.
- Integrate alerting with the existing Prometheus and Grafana monitoring stack.
- Route alerts to email recipients in a maintainable way.
- Reduce operational complexity by centralizing visualization and notification logic.

## Considered Options

### Option 1: Dashboard-only monitoring

Rely on Grafana dashboards without automated notifications.

**Pros**
- Simple to maintain.
- No email or notification configuration required.
- No credential handling required.

**Cons**
- Requires manual dashboard inspection.
- Incidents may go unnoticed for too long.
- Does not provide proactive operational response.
- Not sufficient for critical availability or performance issues.

### Option 2: Alertmanager with SMTP configuration

Use Prometheus Alertmanager to send email notifications through SMTP.

**Pros**
- Alertmanager is designed for alert routing and notification.
- Integrates naturally with Prometheus alert rules.
- Supports grouping, silencing, and routing policies.

**Cons**
- Initial configuration required SMTP credentials in alertmanager.yml.
- Higher risk of exposing sensitive email credentials.
- Adds another component to deploy and maintain.
- Duplicates part of the alerting workflow already available in Grafana.

### Option 3: Grafana alerting with provisioned contact points

Use Grafana-managed alerting, provisioned contact points, notification policies, and SMTP settings configured through environment variables.

**Pros**
- Keeps alerting integrated with the existing Grafana setup.
- Avoids hardcoding SMTP passwords in repository files.
- Supports reproducible alert rules and notification routing.
- Automatically configures alerts when the system starts.
- Reduces the number of alerting components to maintain.

**Cons**
- Depends on Grafana availability for alert delivery.
- Requires Grafana provisioning files to be maintained.
- SMTP configuration still depends on a valid external email provider.

## Decision Outcome

We adopted **Grafana-managed email alerting with provisioned configuration**.

The previous Alertmanager-based email configuration was removed to avoid storing Gmail SMTP credentials directly in repository configuration files.

Grafana SMTP settings are configured in docker-compose.yml, while the SMTP password is provided through the SMTP_PASSWORD environment variable:

GF_SMTP_PASSWORD: ${SMTP_PASSWORD}


Alert contact points, notification policies, and alert rules are provisioned as code under:

monitoring/grafana/provisioning/alerting/


## Implementation Details

### SMTP Configuration

Grafana is configured to send email notifications using Gmail SMTP.

The email account and SMTP settings are defined in docker-compose.yml, while the password is injected through an environment variable instead of being committed to the repository.

### Contact Points

The email contact point is defined in:

monitoring/grafana/provisioning/alerting/contactpoint.yml


It routes notifications to:

tpsoftware03@gmail.com


The same file also defines a webhook contact point for class or integration scenarios.

### Notification Policies

Notification routing is defined in:

monitoring/grafana/provisioning/alerting/notificationpolicies.yml


Alerts are grouped by:

- alertname
- severity

The policy defines grouping and repetition behavior through group_wait, group_interval, and repeat_interval.

### Alert Rules

Grafana alert rules are defined in:

monitoring/grafana/provisioning/alerting/rules.yml


Implemented Grafana alerts include:

- API DOWN GRAFANA
- HIGH ERROR RATE
- HIGH FORECAST LATENCY

These alerts are based on Prometheus metrics and Prometheus alert state.

Prometheus-side alert rules are defined in:

monitoring/prometheus/rules/alerts.yml


They include API availability, error rate, latency, CPU, and memory alerts.

## Decision Outcome

The selected approach provides automated email notifications while keeping sensitive credentials outside version-controlled configuration files.

By using Grafana provisioning, alerting configuration becomes reproducible across environments and is automatically loaded when the monitoring stack starts.

### Consequences

#### Positive

- Automated notification of API outages and performance degradation.
- SMTP password is not hardcoded in repository files.
- Alerting configuration is reproducible and version-controlled.
- Grafana centralizes dashboards, alert rules, contact points, and notification routing.
- Fewer operational components are required compared with using Alertmanager.

#### Negative

- Alert delivery depends on Grafana and the SMTP provider.
- Incorrect or missing SMTP_PASSWORD configuration prevents email delivery.
- Grafana provisioning files require maintenance as alert rules evolve.
- Email alerting can generate noise if thresholds are not tuned correctly.
