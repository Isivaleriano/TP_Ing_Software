# Automated CD to EC2 with GitHub Actions

## Context and Problem Statement

With the CI pipeline working and the Docker image being published to `ghcr.io`, the process of updating the service on EC2 was manual: connect via SSH, pull the image, and restart the application.

This created friction and potential for human error on every deployment.

Additionally, as the project evolved to include multiple services managed through Docker Compose (API, Prometheus, Grafana, webhook receiver and node exporter), a deployment mechanism was needed that could update the API while preserving the monitoring infrastructure and persistent data.

## Decision Drivers

* Need to automate the deployment process.
* Need to reduce manual intervention during releases.
* Need to avoid human errors during service updates.
* Need to ensure deployments only occur after successful builds.
* Need to align the project with Continuous Delivery (CD) practices.
* Need to preserve monitoring services and persistent data during deployments.

## Considered Options

* Continue performing deployments manually via SSH.
* Automate deployments by recreating containers using Docker commands.
* Automate deployments using GitHub Actions and Docker Compose.

## Decision Outcome

Chosen option: "Automate deployments using GitHub Actions and Docker Compose", because it allows the API to be updated automatically while preserving the monitoring infrastructure and persistent volumes.

A deploy job was added to the `docker_publish.yml` workflow using `appleboy/ssh-action` to connect to the EC2 instance via SSH and execute deployment commands remotely.

The deploy job only runs if the Docker build and publish process completes successfully, ensuring that a broken image is never deployed.

Access credentials (IP address, user and SSH key) are stored as GitHub Actions secrets:

* `EC2_HOST`
* `EC2_USER`
* `EC2_SSH_KEY`

The deploy sequence executed on the EC2 instance is:

```bash
docker compose pull api
docker compose up -d --no-deps api
docker compose up -d prometheus grafana webhook-receiver node-exporter
docker image prune -f
```

This approach updates the API service while keeping the monitoring stack running and preserving persistent volumes.

### Consequences

* Good, because every merge to `main` automatically updates the production service.
* Good, because deployments only occur if tests and build steps pass successfully.
* Good, because the entire process is traceable through GitHub Actions logs.
* Good, because manual deployment steps are eliminated.
* Good, because monitoring services remain available during deployments.
* Good, because Grafana and Prometheus data stored in persistent volumes are preserved.

### Confirmation

The deployment process was integrated into GitHub Actions and automatically updates the EC2 environment after a successful build and image publication.

The deployment updates the API service while preserving the monitoring infrastructure and persistent data.

## Pros and Cons of the Options

### Manual deploy via SSH

* Good, because it does not introduce additional pipeline complexity.
* Bad, because it is prone to human error.
* Bad, because it is not aligned with the Continuous Delivery practices required by the project.

### Automate deployments by recreating containers using Docker commands

* Good, because deployments can be automated.
* Good, because updates can be executed remotely.
* Bad, because containers are recreated during each deployment.
* Bad, because service data and monitoring configuration may be affected if persistent storage is not handled correctly.

### Automate deployments using GitHub Actions and Docker Compose

* Good, because deployments are executed automatically.
* Good, because only successful builds are deployed.
* Good, because deployment activity is traceable through workflow logs.
* Good, because monitoring services and persistent volumes are preserved.
* Good, because the deployment process is integrated with the project infrastructure.

## More Information

The deployment process uses `appleboy/ssh-action` to establish the SSH connection and execute deployment commands remotely on the EC2 instance.

Docker Compose is used to manage the application and monitoring services as a single deployment unit.
