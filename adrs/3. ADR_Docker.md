# Docker Based Application Packaging

## Co and Problem Statement

The project started from a Docker example provided in class. That example contained a simple FastAPI application inside a py-app folder and exposed a basic /health endpoint.

While useful for learning how to build and run a Python API inside a container, the example did not match the actual project structure. The project had already evolved into a versioned Oil & Gas Forecast API organized under api/v1.

The team needed a reproducible way to package and run the real application, independent from local Python versions and machine specific configuration. As the project grew, the same containerized approach also needed to support supporting services such as monitoring, database, orchestration, and BI tools.

## Decision Drivers

- Run the FastAPI application in an isolated and reproducible environment.
- Adapt the class Docker example to the actual project structure.
- Avoid depending on local Python installations.
- Provide a simple startup command for the API.
- Support future deployment to cloud infrastructure.
- Keep service configuration explicit and version controlled.

## Considered Options

### Option 1: Run the API directly on the host machine

Install Python dependencies locally and run the API with Uvicorn on the developer or server machine.

**Pros**
- Simple for quick local development.
- No container tooling required.
- Easier to inspect local files during early development.

**Cons**
- Depends on local Python and dependency versions.
- Harder to reproduce across machines.
- More error prone for deployment.
- Does not provide isolation from the host environment.
- Does not scale well as additional services are added.

### Option 2: Keep the original class Docker example

Use the original Docker setup based on the example py-app structure.

**Pros**
- Already available from the class material.
- Provides a basic working Docker reference.
- Useful as a starting point.

**Cons**
- Does not match the project structure.
- Runs the example application instead of the actual API.
- Does not support the api/v1 versioned module path.
- Would require maintaining unused example files.

### Option 3: Custom Dockerfile and Docker Compose setup

Create a Dockerfile adapted to the real FastAPI API and use Docker Compose to run the API together with supporting services.

**Pros**
- Packages the actual application structure.
- Provides reproducible execution across environments.
- Supports deployment through a container image.
- Allows multiple services to run as a coordinated stack.
- Integrates naturally with Prometheus, Grafana, PostgreSQL, Airflow, and Metabase.

**Cons**
- Requires Docker configuration maintenance.
- Adds container build time.
- Requires developers and deployment environments to have Docker available.
- Multi service Compose setups introduce dependency and networking complexity.

## Decision Outcome

We adopted **a custom Dockerfile for the FastAPI API and Docker Compose for multi service execution**.

The API image is built from:

api/v1/Dockerfile


The Dockerfile uses python:3.13-slim as the base image, installs dependencies from api/v1/requirements.txt, copies the project api package into the container, exposes port 8000, and starts the application with Uvicorn:

uvicorn api.v1.main:app --host 0.0.0.0 --port 8000


This allows the container to run the real Oil & Gas Forecast API instead of the original class example.

## Implementation Details

### API Container

The API container is configured to:

- Use a lightweight Python base image.
- Install only the dependencies required by the API.
- Preserve the project module structure under /app/api.
- Expose the API on port 8000.
- Start the application from api.v1.main:app.

### Docker Compose

The project uses:

docker-compose.yml


to run the application stack as a coordinated set of services.

Configured services include:

- api
- prometheus
- grafana
- webhook-receiver
- node-exporter
- postgres
- airflow-init
- airflow-scheduler
- airflow-webserver
- metabase

Docker Compose also defines persistent volumes for services that require state, such as Prometheus, Grafana, PostgreSQL, and Airflow logs.

### Airflow Image

The Airflow services use a dedicated image built from:

Dockerfile.airflow


This image extends apache/airflow:2.9.1 and installs the dependencies needed to run dbt with PostgreSQL from inside Airflow tasks.

### Deployment Compatibility

The API service can be built locally or pulled from GitHub Container Registry through the API_IMAGE variable.

This supports both local development and automated deployment workflows using Docker Compose.

## Decision Outcome

The selected approach provides a reproducible containerized runtime for the API and a consistent way to run the broader project infrastructure.

It also prepares the system for cloud deployment, monitoring, orchestration, and BI integration without requiring each component to be installed manually on the host machine.

### Consequences

#### Positive

- The API runs independently from local Python installations.
- The container matches the real api/v1 project structure.
- Developers can start the stack with Docker Compose.
- Supporting services are defined in version controlled configuration.
- The setup is compatible with CI/CD and EC2 deployment.
- Service networking and dependencies are easier to manage through Compose.

#### Negative

- Docker and Docker Compose are required to run the full stack.
- Image builds add time to the development workflow.
- Compose configuration must be maintained as services evolve.
- Local resource usage increases when running all services together.
- Environment variables and secrets must be managed carefully.
