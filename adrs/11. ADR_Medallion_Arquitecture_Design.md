# Medallion Architecture Design

## Context and Problem Statement

The project requires a data platform capable of ingesting, transforming, validating, and serving oil and gas production data for analytical use cases.

The source datasets include monthly production records and complementary well information. These datasets must be preserved in their original form, cleaned and standardized, and then modeled for reporting and business intelligence.

Without a layered architecture, raw data, cleansing logic, and analytical modeling would be mixed together. This would make the pipeline harder to audit, test, reprocess, and evolve.

## Decision Drivers

- Preserve raw source datasets for traceability and reprocessing.
- Separate ingestion, cleansing, validation, and analytical modeling responsibilities.
- Support reproducible transformations with dbt.
- Enable data quality checks before data reaches analytical consumers.
- Provide a clear path from raw public datasets to business-ready tables.
- Support future orchestration, monitoring, and governance.
- Align with modern Data Engineering practices and the project requirements.

## Considered Options

### Option 1: Single-layer architecture

Store raw, cleaned, and analytical data in the same layer.

**Pros**
- Simple initial implementation.
- Fewer datasets to manage.
- Lower storage usage.

**Cons**
- Poor separation of responsibilities.
- Harder to audit and reprocess data.
- Data quality checks are harder to isolate.
- Raw source data can be overwritten or mixed with transformed data.
- Does not scale well as the pipeline grows.

### Option 2: Direct transformation into a star schema

Load source datasets and transform them directly into fact and dimension tables, without an intermediate cleansing layer.

**Pros**
- Faster path to analytical tables.
- Less intermediate modeling work.
- Simpler pipeline at first.

**Cons**
- No dedicated layer for standardized source data.
- Harder to debug transformation issues.
- Data quality issues may reach the analytical model too late.
- Reduces traceability between source records and final outputs.
- Makes future reprocessing and governance more difficult.

### Option 3: Medallion Architecture

Organize the pipeline into Bronze, Silver, and Gold layers.

**Pros**
- Clear separation between raw, cleaned, and analytical data.
- Preserves source data unchanged in Bronze.
- Enables validation and standardization in Silver.
- Provides business-ready tables in Gold.
- Improves lineage, maintainability, and auditability.
- Works naturally with dbt models and Airflow orchestration.

**Cons**
- Requires more implementation effort.
- Introduces additional datasets and dependencies.
- Uses more storage than a single-layer approach.

## Decision Outcome

We adopted a **Medallion Architecture composed of Bronze, Silver, and Gold layers**.

This architecture provides a clear path from official source datasets to analytical tables while keeping each stage of the pipeline focused on a specific responsibility.

Official source datasets

Bronze: raw ingested data

Silver: cleaned and standardized data

Gold: dimensional analytical model

## Layer Responsibilities

### Bronze Layer

The Bronze layer stores the source datasets exactly as they are received.

Implemented datasets:

- bronze.wells_production
- bronze.listed_wells

Responsibilities:

- Preserve official raw data.
- Support reproducibility and auditing.
- Act as the source of truth for downstream transformations.

No cleansing, deduplication, type conversion, or business transformation is applied in this layer.

### Silver Layer

The Silver layer standardizes and cleans the source data while preserving a source-aligned structure.

Implemented models:

- silver_wells_production
- silver_listed_wells

Responsibilities:

- Normalize strings and business attributes.
- Convert data types.
- Remove duplicate records.
- Standardize dates and monthly production periods.
- Replace or handle null values where appropriate.
- Apply data quality validations before analytical modeling.

The Silver layer provides reliable, reusable datasets for the Gold layer.

### Gold Layer

The Gold layer provides business-ready analytical tables.

Implemented models:

- `fact_production`
- `dim_well`
- `dim_company`
- `dim_location`
- `dim_date`

Responsibilities:

- Implement the analytical dimensional model.
- Separate production measures from descriptive dimensions.
- Generate surrogate keys.
- Preserve fact table grain.

The fact table grain is one production record per well per month, represented by the business key:
(idpozo, anio, mes)


## Data Modeling Decisions

### Integration Key

The datasets are integrated using:
idpozo

This identifier connects production records with well-related information and supports lineage from the source datasets to the analytical model.

### Join Strategy

Some production records may not have a matching record in every dimension.

To avoid losing valid production records, the Gold layer uses `LEFT JOIN` from production records to dimensions and assigns default surrogate keys where needed.

### Slowly Changing Dimensions

The project uses **SCD Type 1** for dimensional attributes.

Historical tracking of attribute changes is not available in the source datasets. For the current analytical requirements, overwriting dimensional attributes keeps the model simpler and avoids unnecessary complexity.

## Decision Outcome

The selected architecture improves traceability, maintainability, and data quality by separating the pipeline into clearly defined layers.

Bronze preserves raw data, Silver prepares trustworthy standardized datasets, and Gold exposes a dimensional model designed for analytical consumption.

### Consequences

#### Positive

- Clear separation between ingestion, cleansing, and analytics.
- Raw data remains available for auditing and reprocessing.
- Data quality checks can be applied before analytical consumption.
- dbt models are easier to organize, test, and document.
- The Gold layer provides a stable interface for reporting and BI tools.
- The pipeline can be orchestrated and monitored more easily.

#### Negative

- More datasets must be maintained across layers.
- Pipeline dependencies become more explicit and must be managed carefully.
- Additional storage is required to preserve Bronze, Silver, and Gold outputs.
- Changes to source structures may require updates in multiple layers.
