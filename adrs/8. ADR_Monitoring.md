# Monitoring and Observability with Prometheus and Grafana

## Context and Problem Statement

As the API evolved, ensuring its availability was no longer sufficient. The team needed visibility into system health, performance, usage patterns, and infrastructure behavior.

Without observability tools, it was difficult to detect failures, diagnose performance issues, monitor resource consumption, or understand how clients interacted with the API.

A monitoring solution was required to collect metrics and visualize system behavior in real time.

## Decision Drivers

- Monitor API availability and reliability.
- Track latency and performance over time.
- Understand API usage patterns.
- Monitor infrastructure resources.
- Improve troubleshooting capabilities.
- Establish observability practices for the system.

## Considered Options

### Rely only on application logs

Use logs as the sole mechanism for monitoring and troubleshooting.

### Implement Prometheus and Grafana

Use Prometheus for metrics collection and Grafana for dashboard visualization.

### Use external SaaS monitoring solutions

Adopt third-party observability platforms.

## Decision Outcome

The team decided to implement an observability stack based on Prometheus and Grafana.

Prometheus was selected because it is specifically designed for collecting and storing time-series metrics. Grafana was selected due to its visualization capabilities and native integration with Prometheus.

The monitoring architecture follows an end-to-end pipeline:

1. The API exposes metrics through the `/metrics` endpoint.
2. Prometheus periodically scrapes those metrics.
3. Grafana queries Prometheus and visualizes the collected data.

## Dashboard as Code

Initially, dashboards were created directly using the Grafana UI. However, dashboard configurations were lost whenever Docker volumes were removed and recreated.

To ensure reproducibility and portability, dashboards were migrated to a "Dashboard as Code" approach.

Dashboards are designed using Grafana UI, exported as JSON files, and version-controlled within the repository.

This approach guarantees:

- Reproducible environments.
- Version control of visualizations.
- Easier collaboration among team members.
- Disaster recovery after infrastructure recreation.

## Custom Metrics

Default HTTP metrics were insufficient to capture business behavior, so custom metrics were implemented.

### Histograms

- `forecast_duration_seconds`
- `wells_duration_seconds`

These metrics measure request latency and allow the computation of percentiles such as p50, p95, and p99.

### Counters

- `forecasts_total`
- `wells_total`

These metrics track API usage and distinguish successful and failed requests through labels.

## Dashboard Structure

### API Availability

This section monitors service reliability:

- Service status (UP/DOWN)
- Uptime over the last 24 hours
- Error rate
- HTTP errors over time

### Latency

This section evaluates endpoint performance:

- p50 latency
- p95 latency
- p99 latency
- Current p95 versus KPI
- Average latency
- SLA compliance

Percentiles provide a better representation of user experience than averages alone.

### Traffic

This section provides visibility into system usage:

- Requests per second per endpoint
- Total forecasts generated
- Total wells generated
- Total requests
- Success versus error distribution

### System Resources

This section correlates application behavior with infrastructure usage:

- CPU utilization
- Memory consumption
- Open file descriptors

## Consequences

### Positive Consequences

- Increased visibility into system behavior.
- Faster troubleshooting of incidents.
- Better understanding of API usage patterns.
- Improved operational awareness.
- Reproducible dashboards through version control.

### Negative Consequences

- Additional infrastructure complexity.
- Monitoring components require maintenance.
- Metrics and dashboards must evolve with the system.

## Confirmation

The monitoring stack successfully provides real-time visibility into API health, performance, business metrics, and infrastructure resources. Version-controlled dashboards ensure reproducibility across environments.

## Pros and Cons of the Options

### Rely only on application logs

**Pros**

- Simple implementation.
- No additional infrastructure required.

**Cons**

- Limited visibility into system behavior.
- Difficult to analyze trends over time.
- No centralized dashboards.

### Implement Prometheus and Grafana

**Pros**

- Industry-standard monitoring solution.
- Powerful visualization capabilities.
- Supports custom metrics.
- Easily integrates with containerized environments.

**Cons**

- Requires additional infrastructure.
- Monitoring configuration must be maintained.

### Use external SaaS monitoring solutions

**Pros**

- Managed infrastructure.
- Advanced features and integrations.

**Cons**

- Additional costs.
- Reduced control over monitoring data.
