# Data Warehouse Dimensional Model

## Context and Problem Statement

The project implements a Medallion Architecture for oil and gas production data. Within this architecture, the Gold layer must provide business-ready analytical tables for reporting, business intelligence, and future forecasting use cases.

The available datasets contain monthly production records and descriptive information about wells, companies, locations, and production periods. Analytical users need to query production and injection metrics by well, company, location, and time.

Without a dimensional Data Warehouse model, analytical queries would depend directly on cleaned source-aligned tables. This would increase query complexity, duplicate business logic, and make reporting less consistent.

## Decision Drivers

- Provide a clear analytical model in the Gold layer.
- Support reporting and BI queries over production metrics.
- Define and document the fact table grain.
- Separate measurable facts from descriptive dimensions.
- Use surrogate keys for stable analytical joins.
- Keep the model understandable for downstream consumers.
- Align with the project requirement to implement a star schema.
- Support dbt documentation, tests, and lineage.

## Considered Options

### Option 1: Single denormalized analytical table

Store production metrics and all descriptive attributes in one wide table.

**Pros**
- Simple to query.
- Fewer joins required.
- Fast initial implementation.

**Cons**
- High redundancy across repeated descriptive attributes.
- Harder to maintain as attributes evolve.
- Business logic would be duplicated in a single large model.
- Does not clearly separate facts from dimensions.
- Does not satisfy the dimensional modeling requirement.

### Option 2: Snowflake schema

Normalize dimensions into additional related subdimensions.

**Pros**
- Reduces redundancy inside dimensions.
- Provides stronger normalization.
- Can be useful for very large or complex hierarchies.

**Cons**
- More joins are required for analytical queries.
- Harder for BI users to understand.
- Adds unnecessary complexity for the current dataset size and scope.
- Increases maintenance effort without significant analytical benefit.

### Option 3: Star schema

Store production and injection metrics in a central fact table and descriptive attributes in surrounding dimensions.

**Pros**
- Clear separation between facts and dimensions.
- Well suited for BI and reporting workloads.
- Easier to understand than a snowflake schema.
- Reduces duplication compared with a fully denormalized table.
- Supports surrogate keys, lineage, documentation, and data quality tests.
- Aligns with the project requirements.

**Cons**
- Requires dimension generation before fact table creation.
- Requires additional joins in analytical queries.
- Introduces surrogate key management.

## Decision Outcome

We adopted a star schema in the Gold layer as the Data Warehouse dimensional model.

The model is centered on monthly well production records and exposes dimensions for well, company, location, and date analysis.

                 dim_well
                    |
dim_company -- fact_production -- dim_location
                    |
                 dim_date

## Dimensional Model

### Fact Table

The central fact table is:
fact_production

Grain:
One row per well per production month.

Business key:
(idpozo, anio, mes)

Surrogate key:
- production_sk

Foreign keys:
- well_sk
- company_sk
- location_sk
- date_sk

Measures:
- prod_pet
- prod_gas
- prod_agua
- iny_agua
- iny_gas
- iny_co2
- iny_otro
- tef

### Dimensions

#### dim_well

Contains one record per well with descriptive well attributes.

Key fields:

- well_sk
- idpozo
- tipopozo
- tipoextraccion
- tipoestado
- formacion
- formprod
- tipo_de_recurso
- sub_tipo_recurso
- clasificacion
- subclasificacion
- vida_util
- coordenadax
- coordenaday

The dimension includes an UNKNOWN record with well_sk = 0 to preserve fact records when a well cannot be matched.

#### dim_company

Contains one record per operating company.

Key fields:

- company_sk
- idempresa

#### dim_location

Contains distinct geographical and operational location combinations.

Key fields:

- location_sk
- provincia
- cuenca
- areayacimiento
- areapermisoconcesion

#### dim_date

Contains one record per production year-month.

Key fields:

- date_sk
- anio
- mes
- quarter

The date_sk is generated in YYYYMM format using:

text
anio * 100 + mes


## Data Modeling Decisions

### Fact Grain

The fact table grain is one row per well per month.

This grain matches the source production records and is validated through the unique combination:

text
(idpozo, anio, mes)


### Surrogate Keys

The model uses surrogate keys for fact and dimension tables:

- production_sk
- well_sk
- company_sk
- location_sk
- date_sk

Surrogate keys provide stable joins for analytical consumers and allow the model to handle missing or incomplete natural keys more explicitly.

### Join Strategy

The fact table uses LEFT JOIN from production records to dimensions.

This approach preserves production records even when some dimensional attributes are incomplete or missing. Where applicable, missing dimension matches are handled with default surrogate keys.

### Slowly Changing Dimensions

The Data Warehouse uses **SCD Type 1** for dimensional attributes.

The source datasets do not provide reliable historical tracking of attribute changes. For the current analytical requirements, overwriting descriptive attributes is simpler and sufficient.

### Analytical Consumption

Analytical consumers should query the Gold dimensional model instead of directly querying Bronze or Silver tables.

Bronze and Silver remain internal pipeline layers, while Gold acts as the stable analytical interface.

## Decision Outcome

The selected star schema provides a clear and maintainable Data Warehouse model for oil and gas production analytics.

It supports production analysis by well, company, location, and date while keeping business logic centralized in dbt models and documented through dbt metadata and tests.

### Consequences

#### Positive

- Clear separation between measurable facts and descriptive dimensions.
- Stable Gold layer interface for BI and reporting.
- Documented fact table grain and business key.
- Reduced redundancy compared with a single wide table.
- Better support for dbt tests, documentation, and lineage.
- Easier extension with new dimensions or measures.

#### Negative

- Dimension tables must be built before the fact table.
- Analytical queries require joins between facts and dimensions.
- Surrogate key generation must remain consistent.
- SCD Type 1 does not preserve historical versions of dimensional attributes.
