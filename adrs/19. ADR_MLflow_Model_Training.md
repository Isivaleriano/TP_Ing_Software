# MLflow-Based Model Training and Registry for Production Forecasting

## Context and Problem Statement

It was required a supervised machine learning pipeline that predicts a well's next month's production — oil first, gas later — from `gold.training_dataset_production`. Beyond fitting a model, the pipeline needed: a way to compare multiple candidate configurations (model family and hyperparameters) reproducibly across training runs; a way for the serving code (`ml/predict.py`, the `/forecast/ml` API endpoint) to always retrieve "the
current best model" without hardcoding a file path or version number; and an evaluation protocol that does not leak future information, since the features are lagged monthly production values and the label is next month's production for the same well.

## Decision Drivers

- Reproducibility: every training run's parameters and metrics must be recoverable later, not just the final chosen model.
- Decoupling: serving code should resolve "the current best model" symbolically, so retraining and promoting a new version doesn't require a code change or redeploy.
- No label leakage: `gold.feature_store_production` intentionally keeps rows with an unknown (`NULL`) target — each well's most recent month, kept there so it can be used for live inference (see `18. ADR_Feature_Store.md`). Training must never see those rows; only rows with a real, already-realized outcome can be used to fit or evaluate a model.
- No temporal leakage: production is highly autocorrelated month to month, so a random train/test split would overstate accuracy; evaluation must simulate genuinely future, unseen periods.
- Multi-target extensibility: the same training code should support additional target variables (oil now, gas added afterwards) with minimal duplication.
- Reuse the project's existing Docker Compose conventions instead of adding a new managed/cloud dependency.

## Considered Options

### Manual model versioning (pickled files + naming convention)

Save every trained scikit-learn pipeline as `model_<date>.pkl` on disk, track parameters and metrics in a spreadsheet or JSON log next to it, and point `predict.py` at whichever path an environment variable happens to reference.

### Self-hosted MLflow: Tracking Server + Model Registry (chosen)

Run an MLflow Tracking Server (SQLite backend store, local filesystem artifact store) as the `mlflow` service in `docker-compose.yml`. Every candidate configuration is logged as an MLflow run (parameters, validation metrics, and the fitted pipeline as an artifact); the best pipeline per target is registered under a named model and promoted with a `Champion` alias in the Model Registry.

### Managed experiment-tracking platform (e.g. Weights & Biases, Databricks)

Use a hosted third-party service for experiment tracking and model registry.

## Decision Outcome

Chosen option: **self-hosted MLflow**, combined with five additional design decisions in how the training pipeline (`ml/train.py`) and serving code (`ml/predict.py`) use it:

### 1. Train exclusively from `gold.training_dataset_production`, never `feature_store_production`

`load_dataset()` in `ml/train.py` reads only `gold.training_dataset_production` (`SELECT * FROM gold.training_dataset_production`) — it never queries `gold.feature_store_production` directly, and never re-implements a "does this row have a known target" filter in Python. This was a deliberate anti-leakage decision, distinct from the temporal split below: `feature_store_production` keeps a row per well per month *including* each well's most recent month, whose target is `NULL` precisely because it's meant for live inference (`18. ADR_Feature_Store.md`). If `train.py` read from that table and filtered in Python, any bug, a forgotten condition, or `NaN`-handling quirk downstream (e.g. an imputer silently treating a `NULL` target as a numeric value) could let an inference-only, label-less row leak into training or evaluation. By instead depending on `gold.training_dataset_production` — a dbt model whose sole job is to apply that filter once, tested with `not_null` on both target columns in `schema.yml` — the "only rows with a real, already-realized outcome" guarantee is enforced at the data layer, in one place, instead of being re-derived (and possibly gotten wrong) in every consumer. This is also why extending to gas only required updating the dbt filter (`target_prod_pet_next_month IS NOT NULL AND target_prod_gas_next_month IS NOT NULL`) rather than touching `ml/train.py`'s loading logic at all.

### 2. Temporal train / validation / test split, not random

`temporal_split()` builds a calendar `period` (`anio * 100 + mes`) per row, sorts the distinct periods, and holds out the last 6 months as test and the preceding 6 months as validation; everything older is train. Grid search picks the best configuration by validation RMSE; the winner is then refit on train + validation combined and scored once, on the held-out test months, for the reported metric. This mirrors how the model is actually used — predicting months that haven't happened yet — instead of a random holdout that would let the model see rows from the same or later months as ones used to predict them.

### 3. Two model families, grid-searched, per target

For each target, `train_target()` grid-searches `HistGradientBoostingRegressor` (5 configurations) and `RandomForestRegressor` (4 configurations), logging every run as its own MLflow run under the target's experiment (`oil_production_forecasting` / `gas_production_forecasting`). Two different model families were kept, rather than committing to one upfront, because they make different tradeoffs on this feature set (categorical well/location attributes plus numeric lags): gradient boosting via native missing-value handling versus a random forest with explicit median imputation. Keeping both in the same grid search means the final champion is chosen empirically by validation RMSE, not by a family picked in advance.

### 4. Alias-based promotion (`Champion`)

Once the best configuration is refit and evaluated on test, it is logged again and registered under a fixed model name; "Champion".

### 5. One model per commodity, one shared training function

Oil and gas are trained as two independent regressors — `oil_production_forecaster` and `gas_production_forecaster` — rather than a single multi-output model, each with its own MLflow experiment. `train.py` factors the shared logic (dataset loading, temporal split, grid search, evaluation, registration) into `train_target(target, experiment_name, model_name, ...)`, called once per commodity from `main()`. Both models therefore go through the identical split strategy, param grids and champion-selection logic.

## Consequences

### Positive Consequences

- Sourcing training data exclusively from `gold.training_dataset_production` means the "only real, known outcomes" guarantee lives in one tested dbt model instead of being re-implemented (and possibly re-broken) in `ml/train.py`; adding the gas target required zero changes to the training script's data-loading code.
- Every candidate run (5 HGB + 4 RF configurations, per target) is logged with its parameters and validation metrics, so past experiments stay comparable and auditable from the MLflow UI (`http://localhost:5001`) instead of living only in a terminal's scrollback.
- `predict.py` and the `/forecast/ml` API endpoint never hardcode a model version — they always load `@Champion`, so retraining and promoting a new model requires no code deploy.
- The temporal split produces a test-set metric that reflects genuine forecasting performance instead of an optimistic random-holdout estimate.

### Negative Consequences

- Both grid searches run sequentially inside one process, and `RandomForestRegressor(n_jobs=-1)` saturates every available CPU core per run.
- Oil and gas are modeled independently, so no shared representation or cross-target signal is exploited between them, even though the two are physically correlated for the same well.

## Pros and Cons of the Options

### Manual model versioning

**Pros**

- Zero extra infrastructure to run or maintain.

**Cons**

- No structured, queryable comparison across runs.
- Serving code must know which file/version is "current" — promotion means editing a path or environment variable, not an atomic operation.
- No built-in audit trail linking an artifact back to the parameters that produced it.

### Self-hosted MLflow

**Pros**

- Every run's parameters, metrics and artifact are queryable from one UI.
- Registry + aliases decouple training from serving.
- Runs as one more service inside the project's existing Docker Compose setup.

**Cons**

- One more service to run and maintain (`mlflow` container).
- SQLite backend store is a single point of failure, not built for concurrent access at scale.

### Managed experiment-tracking platform

**Pros**

- No self-hosting; built-in collaboration and sharing features.

**Cons**

- External dependency and cost disproportionate to a single batch-trained model at coursework scale.
- Data would leave the local/EC2 environment.
