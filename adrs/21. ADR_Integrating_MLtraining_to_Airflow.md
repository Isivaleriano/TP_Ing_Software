# Integrating Machine Learning Training into the Airflow Pipeline

## Context and Problem Statement

Phase 3 requires the machine learning models to be trained automatically as part of the data pipeline.

Until this stage, the ETL process ended after building the Gold layer and validating data quality through dbt tests. Model training had to be executed manually by running `ml/train.py`, making the pipeline only partially automated and preventing reproducible end-to-end executions.

The main architectural decision was how to integrate model training into the existing orchestration pipeline while keeping responsibilities separated and allowing experiment tracking through MLflow.

## Decision Drivers

- Automate the complete pipeline from raw data ingestion to model training.
- Preserve the existing ETL structure implemented with Airflow and dbt.
- Register experiments automatically in MLflow.
- Avoid coupling model training with transformation logic.
- Allow independent evolution of the ETL and Machine Learning components.
- Keep the orchestration simple enough for the project's scope.

## Considered Options

### Execute model training manually

Keep the existing pipeline unchanged and train the models outside Airflow whenever required.

### Integrate model training inside dbt

Embed the machine learning script as part of the dbt transformations.

### Add a dedicated Airflow task after data validation (chosen)

Create a new Airflow task responsible only for executing the machine learning training script after the ETL and data quality validation have completed successfully.

## Decision Outcome

Chosen option: **introduce a dedicated `train_forecasting_models` Airflow task**.

The new task executes `ml/train.py` inside a dedicated Python virtual environment (`ml-env`) after the following pipeline stages complete successfully:

- raw data extraction;
- Bronze loading;
- dbt model execution;
- dbt data quality tests;
- persistence of quality results.

The task communicates with the MLflow Tracking Server to automatically create experiments, register runs and store trained models.

Machine learning dependencies are isolated from the dbt environment by introducing an independent virtual environment (`ml-env`) inside the Airflow container.

## Consequences

### Positive Consequences

- End-to-end pipeline execution becomes fully automated.
- Every successful pipeline execution produces updated machine learning models.
- Experiments are automatically tracked in MLflow.
- Machine learning dependencies remain isolated from dbt dependencies.
- Training can be retried independently through Airflow.
- The orchestration remains simple and follows the existing DAG structure.

### Negative Consequences

- Airflow images become larger because they contain both dbt and machine learning environments.
- Pipeline execution time increases due to model training.
- Additional infrastructure resources (disk space and MLflow server) are required.

## Pros and Cons of the Options

### Manual training

**Pros**

- Simplest implementation.
- No additional infrastructure required.

**Cons**

- Pipeline is not fully automated.
- Difficult to reproduce executions.
- Experiments may not be consistently tracked.

### Training inside dbt

**Pros**

- Single execution tool.

**Cons**

- Blurs the separation between data transformation and machine learning.
- Couples dbt with ML libraries.
- Makes dependency management significantly more complex.

### Dedicated Airflow task

**Pros**

- Clear separation between ETL and Machine Learning.
- Fully automated pipeline.
- Native orchestration, retries and monitoring through Airflow.
- Automatic experiment tracking in MLflow.
- Independent dependency management through separate virtual environments.

**Cons**

- Additional Airflow task.
- Slightly more complex Docker image.

## More Information

The implementation introduces:

- a new Airflow task (`train_forecasting_models`);
- a dedicated Python virtual environment (`ml-env`) inside the Airflow container;
- automatic execution of `ml/train.py`;
- integration with the MLflow Tracking Server for experiment logging and model registration;
- execution only after successful completion of ETL, dbt transformations and data quality validation.
