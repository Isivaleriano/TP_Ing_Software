# Data Quality Validation in the Medallion Architecture

## Context and Problem Statement

The project implements a Medallion Architecture composed of Bronze, Silver, and Gold layers for oil and gas production data. While the Bronze layer preserves raw ingested data, the Silver and Gold layers are responsible for delivering cleaned, standardized, and analytics-ready datasets.

Without systematic validation, data issues such as missing values, duplicated records, invalid domain values, or broken relationships between tables could propagate through the pipeline and negatively impact downstream analytics and reporting.

To ensure reliability and trustworthiness of the transformed datasets, a data quality framework was required.

## Decision Drivers

- Improve trust in analytical datasets.
- Detect data inconsistencies early in the pipeline.
- Enforce data integrity across layers.
- Validate dimensional modeling assumptions.
- Provide automated and reproducible quality checks.
- Align with modern Data Engineering best practices.

## Considered Options

### Option 1: Manual validation using SQL queries

Perform ad-hoc validation queries directly in PostgreSQL after each transformation.

**Pros**
- Simple to implement.
- No additional tooling required.

**Cons**
- Difficult to maintain.
- Not reproducible.
- Validation logic remains undocumented.
- High risk of human error.

### Option 2: Custom validation scripts in Python

Implement validation checks through custom Python scripts.

**Pros**
- Flexible.
- Allows complex validations.

**Cons**
- Additional codebase maintenance.
- Validation logic is separated from the transformation layer.
- Less integrated with dbt workflows.

### Option 3: Data Quality Testing with dbt

Implement automated data quality tests directly within dbt using schema definitions and reusable test macros.

**Pros**
- Native integration with transformation models.
- Declarative and maintainable.
- Easy execution through `dbt test`.
- Supports built-in and custom tests.
- Clear documentation of data quality expectations.

**Cons**
- Requires initial setup effort.
- Limited to validations supported by dbt and installed packages.

## Decision Outcome

We adopted **dbt Data Tests** as the primary mechanism for data quality validation.

Quality controls were implemented in both Silver and Gold layers.

### Silver Layer

The Silver layer focuses on cleansing and standardization of source data.

Implemented validations include:

- Uniqueness of well identifiers.
- Non-null checks on critical business attributes.
- Validation of month values (1–12).
- Referential consistency between production records and listed wells.
- Coordinate range validation.
- Non-negative depth validation.
- Fact grain validation using the composite key (`idpozo`, `anio`, `mes`).

### Gold Layer

The Gold layer focuses on dimensional consistency and analytical correctness.

Implemented validations include:

- Surrogate key uniqueness.
- Surrogate key non-null constraints.
- Referential integrity between fact and dimension tables.
- Valid calendar dimensions.
- Fact table grain validation.
- Non-negative production metrics (`prod_pet`, `prod_gas`, `prod_agua`).

## Decision Outcome

The adoption of dbt-based data quality testing provides a scalable, maintainable, and automated validation framework integrated directly into the transformation pipeline.

The selected approach enables early detection of data quality issues while documenting both technical and business assumptions about the datasets.

### Consequences

#### Positive

- Improved confidence in analytical datasets.
- Early detection of data quality issues.
- Automated validation integrated into the transformation workflow.
- Better documentation of business and technical assumptions.
- Easier maintenance and scalability of the data platform.

#### Negative

- Additional maintenance effort for test definitions.
- Some source-system inconsistencies generate warnings that require manual review.
- Slightly longer pipeline execution times due to validation steps.
