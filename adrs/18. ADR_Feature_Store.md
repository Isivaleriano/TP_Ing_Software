# Feature Store for the Production Forecasting Model

## Context and Problem Statement

It was requiered a predictive model for hydrocarbon production. The `Feature Store` sits between the Gold data warehouse and both `Training` and the `Servicio de API`. It was demanded that the feature store must be used during inference, not only to build a training set.

Forecasting next month's oil production for a well needs lag features (production in prior months) and a supervised target (next month's production) that don't exist in `gold.fact_production` as-is. Two implementation questions had to be resolved: how to compute correct lag/target values from a real-world dataset where not every well reports every month, and how to structure the resulting tables so the *same* features serve both training and live inference.

## Decision Drivers

- Lag and target values must reflect actual calendar months, not just "the previous row for this well" — the production dataset has wells with gaps in their monthly reporting.
- The feature store must remain usable for inference: it needs to retain a usable row for the most recent month of every well, not just historical rows with a known label.
- Should reuse the existing dbt/Gold layer infrastructure and its established conventions (table materialization, `schema.yml` tests/docs) rather than introduce a parallel pipeline.
- Should not discard more data than necessary to get correct features — wells with shorter histories are still useful for everything except the lags they can't yet have.

## Considered Options

### SQL window functions (`LAG`/`LEAD` over `idpozo` ordered by `anio, mes`)

The first implementation: `LAG(prod_pet, N) OVER (PARTITION BY idpozo ORDER BY anio, mes)` for each lag, `LEAD(prod_pet, 1)` for the target.

### Self-join on calendar month (chosen)

`feature_store_production` builds a `production_month` date (`make_date(anio, mes, 1)`) per row, then left-joins the same CTE to itself once per lag (`production_month = current_month - interval 'N month'`) and once for the target (`+ interval '1 month'`).

### Python script outside dbt (pandas), feature table built separately

Compute the same lag/rolling features in a standalone Python script reading from `gold.fact_production`, writing the result to its own table via a database client, independent of the dbt project.

## Decision Outcome

Chosen option: **self-join on calendar month**, implemented as two dbt models — `feature_store_production` (every well-month, lag features, and a target that is `NULL` for each well's most recent month) and `training_dataset_production` (the same table filtered to `target_prod_pet_next_month IS NOT NULL`).

The window-function version was tried first and discarded: `LAG`/`LEAD` shift by row position within the partition, not by calendar time. Checked against the real warehouse, 14% have at least one gap in their monthly reporting, so for rows next to a gap, `prod_pet_lag_1` was silently the well's *last reported* month, not necessarily *last calendar* month — wrong, but not absent, so nothing would have caught it. The self-join version fixes this by matching on the actual calendar month, so a lag is only populated when that exact prior month exists.

The Python/pandas option was rejected to keep feature engineering inside the same dbt project as the rest of Gold — one set of conventions, tests and documentation, instead of a second pipeline with its own database access code. Feast was rejected for the same reason as in the project's other infrastructure decisions: a dedicated online/offline feature store is disproportionate for one batch-trained model with no low-latency serving requirement.

### Why two tables, not one filtered table

An early version of `feature_store_production` filtered out any row missing *any* lag column (`prod_pet_lag_1` through `lag_12`, requiring 12 full consecutive prior months). That is enough for a clean training set, but it also throws away every well that doesn't yet have a year of history — and it is exactly the row it is needed for inference that this filter always excludes. Measured against the same warehouse, that filter kept 85% of the rows and would keep proportionally less as more long-range lags are added.

The current design instead keeps every row in `feature_store_production` — including each well's most recent month, where the lag columns may be partially populated and the target is `NULL` because the next month hasn't happened yet — and only drops rows in the separate `training_dataset_production` table, where the single thing that disqualifies a row is not having a known target. Measured the same way, this keeps 98% of rows for training, therefore less potential useful information is lost. 

## Consequences

### Positive Consequences

- `feature_store_production` is usable for inference as-is: a well's latest row is present, with whatever lags it has and `target_prod_pet_next_month = NULL`.
- `training_dataset_production` is a simple, always-correct view for training: anything in it has a real label.
- Lags are calendar-correct even for wells with reporting gaps, instead of silently shifting by an unpredictable number of months.
- Both tables follow the same dbt conventions (materialization, `schema.yml`, `dbt_utils.unique_combination_of_columns` on `(idpozo, anio, mes)`) as the rest of Gold, so they get the same tooling (lineage, docs, tests).

### Negative Consequences

- A well with a short or gappy history still has `NULL` lag columns in both tables (e.g. `prod_pet_lag_12` for a well with under a year of data); the feature store does not impute or flag this.
- Seven self-joins per model (one per lag, plus the target) over the full `fact_production` history is heavier than a single windowed pass; acceptable at this dataset's size but worth revisiting if the warehouse grows substantially.

## Pros and Cons of the Options

### SQL window functions

**Pros**

- Single pass over the data, simplest SQL to write.

**Cons**

- Lag/lead shift by row position, not calendar time — silently wrong for any well with a reporting gap (measured: 14% of wells affected).

### Self-join on calendar month

**Pros**

- Lags are correct even across reporting gaps.
- Stays inside the existing dbt project and its conventions.

**Cons**

- More joins, more compute than a windowed pass.

### Python script outside dbt

**Pros**

- Same engineering trade-offs as the self-join, just expressed in pandas instead of SQL.

**Cons**

- A second pipeline with its own database access, separate from dbt's tests/docs/lineage.

### Dedicated feature store tool (Feast)

**Pros**

- Purpose-built for train/serve consistency, with online/offline store separation.

**Cons**

- Significant new infrastructure for a single batch-trained model with no low-latency requirement.
