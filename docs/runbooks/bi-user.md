# Runbook: BI User

## 1. Purpose and Trigger

### Purpose

This runbook defines the operational procedures required for Business Intelligence (BI) users to access, analyze and validate information published by the Oil & Gas Data Platform.

The BI User is responsible for consuming analytical information, monitoring business metrics and reporting any inconsistencies identified in dashboards or reports.

### When to Execute

#### Scheduled Review

This runbook shall be executed whenever busines users need to review operational or production metrics generated by the platform.

#### Event

This runbook shall be executed when:

- New dashboards are published.
- New business metrics become available.
- New analytical reports are released.

#### Incident

This runbook shall be executed when:

- A dashboard displays incomplete information.
- Business metrics appear inconsistent.
- Data appears outdated.
- A dashboard becomes unavailable.

#### Request

This runbook shall be executed when:

- A business area requests production information.
- Management requests analytical reports.
- Historical production data needs to be analyzed.

---

## 2. Role Owner and Prerequisites

### Role Owner

Business Intelligence User (BI User).

### Required Access

The BI User shall have access to:

- Metabase.
- Published dashboards and reports.
- Approved analytical datasets.

### Required Inputs

The BI User shall have access to:

- Business requirements.
- Operational reporting needs.
- Dashboard URLs.

### Required Tools

- Web browser.
- Metabase.

---

## 3. Procedures

### 3.1 Access Analytical Dashboards

The BI User shall access metabase using approved credentials.

The BI User shall navigate to the dashboard corresponding to the business area or reporting requirement.

---

### 3.2 Review Business Metrics

The BI User shall review the metrics presented in dashboards and reports.

Examples include:

- Oil production.
- Gas production.
- Production trends over time.
- Production by company.
- Production by location.
- Production by well.

The BI User shall use available filters to narrow results when required.

---

### 3.3 Analyze Historical Information

When historical analysis is required, the BI User shall:

1. Select the desired time period.
2. Apply the necessary filters.
3. Compare current and historical results.
4. Export results if necessary.

---

### 3.4 Validate Data Freshness

The BI User shall verify that dashboard information corresponds to the latest available data.

The BI User shall confirm that dashboards reflect recent pipeline executions and expected reporting periods.

---

### 3.5 Report Data Issues

If anomalies are detected, the BI User shall:

1. Document the issue.
2. Capture evidence when possible.
3. Notify the Data Engineer.
4. Provide sufficient context to reproduce the issue.

Examples include:

- Missing data.
- Unexpected values.
- Inconsistent aggregations.
- Empty dashboards.

---

## 4. Validation

### Dashboard Availability Validation

The BI User shall verify that dashboards load successfully.

### Data Availability Validation

The BI User shall verify that expected information is present for the selected reporting period.

### Consistency Validation

The BI User shall compare results against previous reports when applicable.

### Freshness Validation

The BI User shall verify that dashboard information reflects the most recent data available within the platform.

### Expected Results

The BI User shall confirm:

- Dashboards are accessible.
- Data is available for the requested period.
- Metrics appear consistent.
- Information is up to date.
- No visual or calculation errors are present.

---

## 5. Failure Handling

### Dashboard Unavailable

#### Possible Causes

- Metabase unavailable.
- Network connectivity issues.
- Platform maintenance.

#### Resolution

The BI User shall notify the Data Engineer and provide details regarding the affected dashboard.

---

### Missing Data

#### Possible Causes

- Pipeline execution failure.
- Delayed source data publication.
- Data quality issue.

#### Resolution

The BI User shall report the issue to the Data Engineer and provide the affected report and reporting period.

---

### Inconsistent Metrics

#### Possible Causes

- Transformation error.
- Incorrect filtering.
- Data quality issue.

#### Resolution

The BI User shall document the inconsistency and escalate the issue for investigation.

---

### Outdated Information

#### Possible Causes

- Pipeline execution delay.
- Airflow scheduling issue.
- Data source update delay.

#### Resolution

The BI User shall request verification of the latest pipeline execution.

---

## 6. Non-Functional Considerations

### Data Freshness

The platform updates information daily through automated pipeline execution.

Business users should expect data freshness consistent with the latest successful pipeline execution.

### Data Quality

Published dashboards consume only validated analytical datasets.

Data quality controls are executed automatically before data becomes available for reporting.

### Security and Privacy

BI Users have read-only access to analytical information.

No modification of source or transformed datasets is permitted.

### Availability

Dashboard availability depends on the availability of metabase, postgreSQL and the data pipeline.

### Governance

Business metrics exposed through dashboards are managed by the data platform team.

Changes to metrics or calculations must follow the established change management process.

---

## Functional Decision Justification

The platform exposes business information exclusively through the gold layer of the Medallion architecture.

From the BI User perspective, this decision ensures that dashboards consume curated, business-ready datasets rather than raw or intermediate data structures.

This simplifies analysis and reduces the risk of misinterpretation.

---

## Non-Functional Decision Justification

The platform provides read-only access to business users.

From the BI User perspective, this decision protects the integrity of analytical datasets while ensuring broad accessibility to trusted information across the organization.
